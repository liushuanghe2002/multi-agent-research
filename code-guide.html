🗃 共享状态：Agent 之间的"黑板"
State 是整个系统最核心的数据结构，理解它就理解了 Agent 间如何通信
research_agent.py — State 定义
from typing import TypedDict

class State(TypedDict):         # ① 继承 TypedDict，获得类型检查
    topic:    str               # ② 只读字段，由调用者在 invoke() 时传入
    research: str               # ③ Researcher 写入，Analyst 读取
    analysis: str               # ④ Analyst 写入，Summariser 读取
    summary:  str               # ⑤ Summariser 写入，最终结果
📋 字段所有权矩阵
字段名 类型 写入者
topic str 用户研究主题，如 "量子计算最新进展" 调用方输入
research str Tavily 搜索返回的 5 篇文章标题+正文，原始未处理 📚 Researcher
analysis str LLM 对搜索结果的结构化分析：关键事实、冲突信息 📊 Analyst
summary str 面向非专家的最终报告，含 Key Takeaways 💬 Summariser
💡 为什么用 TypedDict？
TypedDict 让 Python 知道字典里有哪些 key，IDE 可以自动补全，mypy 可以做类型检查。LangGraph 也依赖这个结构来验证 Agent 的返回值格式是否正确。
⚠️ 单向写入原则：每个 Agent 只写自己负责的字段，绝不覆盖其他 Agent 的输出。这避免了并发写入冲突，也让数据流向清晰可追溯。
🔄 State 在执行中的变化
invoke 前：
{"topic": "量子计算"}
↓ Researcher 执行后
{"topic": "量子计算", "research": "IBM宣布...Google实现..."}
↓ Analyst 执行后
{"topic":..., "research":..., "analysis": "关键发现：1...."}
↓ Summariser 执行后
{"topic":..., "research":..., "analysis":..., "summary": "..."}
🧠 延伸：LangGraph 的 Annotated State（进阶用法）
进阶示例 — 带累积器的 State
from typing import Annotated
import operator

class AdvancedState(TypedDict):
    topic:    str
    # 用 Annotated + operator.add 让 messages 字段自动累积（而不是覆盖）
    messages: Annotated[list, operator.add]   # ← 这是 LangGraph 的"消息累积器"
    research: str
    analysis: str
🔬 当多个 Agent 并行写入同一字段时，用 Annotated[list, operator.add] 可以把结果追加而不是覆盖。这是 LangGraph 处理并行 Reduce 的核心机制。

Agent 1：Researcher
职责：调用 Tavily 搜索 API，返回原始文章
工具调用型 Agent
research_node()行 31–45
def research_node(state: State) -> dict:
    """
    注意返回类型是 dict，而不是完整 State。
    LangGraph 会把返回的 dict 浅合并（merge）进全局 State。
    """
    tavily = TavilyClient(api_key=TAVILY_API_KEY)  # ①
    results = tavily.search(                              # ②
        state["topic"],                               # 读取用户输入
        max_results=5                                  # 搜索 5 篇文章
    )

    combined_results = ""
    for r in results["results"]:                       # ③
        combined_results += r["title"] + "\n" + r["content"] + "\n\n"

    return {"research": combined_results}                 # ④ 只写自己负责的字段
① TavilyClient 每次调用都新建实例。生产中应该复用（全局变量或依赖注入），避免反复初始化的开销。
② tavily.search() 返回的 JSON 结构：
{"results": [{"title":..., "url":..., "content":...}, ...]}
③ 简单拼接所有文章。注意：这里没有做去重、质量过滤。生产中应加来源可信度评分。
④ 关键：只返回 {"research": ...}，LangGraph 自动 merge 进全局 State，topic/analysis/summary 字段不受影响。
🔧
动手改造：把 max_results=5 改成从 State 读取：state.get("max_results", 5)，然后在 State 类里加 max_results: int 字段，用户就可以控制搜索深度了。

Agent 2：Analyst
职责：用 LLM 分析原始搜索结果，提炼关键见解
LLM 推理型 Agent
analyst_node()行 48–78
def analyst_node(state: State) -> dict:
    groq_client = Groq(api_key=GROQ_API_KEY)

    response = groq_client.chat.completions.create(
        model=MODEL,                                         # ① llama-3.1-8b-instant
        messages=[
            {
                "role": "system",                            # ② 角色提示
                "content": (
                    "You are a research analyst. "
                    "Extract key insights, identify important facts, "
                    "and note any conflicting information."
                ),
            },
            {
                "role": "user",
                "content": (                                    # ③ 把 topic + research 一起传入
                    f"Topic: {state['topic']}\n\n"
                    f"Research:\n{state['research']}"
                ),
            },
        ],
    )

    answer = response.choices[0].message.content         # ④ 取第一个候选输出
    return {"analysis": answer}
① Groq API 接口与 OpenAI 100% 兼容，只需换 base_url 和 api_key，代码无需其他修改即可切换到 GPT-4。
② System Prompt 定义了 Agent 的角色和能力边界。这里要求"客观、找冲突信息"，是高质量 Analyst 的关键。
③ 把前一个 Agent 的输出（research）和原始 topic 一起打包给 LLM。这就是 Agent 间"通过 State 通信"的具体体现。
④ response.choices[0].message.content 是 OpenAI 风格 API 的标准取值方式。如果开启了 n>1，choices 里会有多个候选。
💡
Prompt 工程要点：Analyst 的 System Prompt 要求 "note any conflicting information between sources"（标记来源冲突）。这非常重要——搜索结果可能相互矛盾，让 LLM 显式标记冲突，Summariser 才能给出平衡的报告，而不是只取一家之言。
💬
Agent 3：Summariser
职责：把分析结果转化为非专家可读的结构化报告
输出生成型 Agent
summariser_node()行 81–109
def summariser_node(state: State) -> dict:
    groq_client = Groq(api_key=GROQ_API_KEY)

    response = groq_client.chat.completions.create(
        model=MODEL,
        messages=[
            {
                "role": "system",
                "content": (
                    "You are a technical writer. "             # ① 角色切换！
                    "Write a clear summary a non-expert could understand. "
                    "Include key takeaways at the end."           # ② 要求结构化输出
                ),
            },
            {
                "role": "user",
                "content": (                                    # ③ 只读 analysis，不读原始 research
                    f"Topic: {state['topic']}\n\n"
                    f"Analysis:\n{state['analysis']}"
                ),
            },
        ],
    )

    answer = response.choices[0].message.content
    return {"summary": answer}
① System Prompt 换成了 "technical writer"（技术写作者），而不是 "research analyst"。同一个 LLM，不同的角色，行为完全不同——这正是 Multi-Agent 分工的精髓。
③ 注意 Summariser 没有读取 state["research"]——它只处理 Analyst 精炼后的分析，避免被原始噪音干扰。这是信息分层处理的体现。
🎯
三个 Agent 的 System Prompt 对比：
• Researcher：没有 LLM，只有工具调用
• Analyst："research analyst — objective, find conflicts"（客观分析师）
• Summariser："technical writer — clear, non-expert friendly, key takeaways"（技术写作者）
→ 角色定义 = Agent 能力边界，这是 Prompt 设计最核心的部分。
🕸 LangGraph 图编排：把 Agent 连起来
build_graph() 是整个项目的"指挥中心"，理解它就掌握了 LangGraph 的核心 API
build_graph()行 112–128
def build_graph():
    graph = StateGraph(State)         # ① 创建图，绑定 State 类型

    # ② 注册节点：字符串名称 → Python 函数
    graph.add_node("research",   research_node)
    graph.add_node("analyst",    analyst_node)
    graph.add_node("summariser", summariser_node)

    # ③ 添加有向边：定义执行顺序
    graph.add_edge(START,          "research")   # 从起点进入 Researcher
    graph.add_edge("research",    "analyst")    # Researcher → Analyst
    graph.add_edge("analyst",     "summariser") # Analyst → Summariser
    graph.add_edge("summariser",  END)           # Summariser → 终点

    return graph.compile()              # ④ 编译为可执行的 Runnable
① StateGraph(State)：把 State 类型"绑定"到图上，LangGraph 会用这个类型来验证每个节点的输入输出，防止传错字段。
② add_node("name", fn)：节点名称是字符串，函数是节点逻辑。名称在 add_edge() 里引用，函数可以是任何 callable（lambda、类方法、闭包）。
③ add_edge(A, B)：声明执行顺序，"A 完成后执行 B"。顺序与代码里 add_edge 的先后无关，只和边的定义有关。
④ compile()：把图"冻结"成 Runnable 对象，支持 .invoke()/.stream()/.astream() 等接口，也可以加 checkpointer 实现断点续跑。
🔀 进阶：条件边（Conditional Edges）
如果 Analyst 认为搜索结果质量太差，想让 Researcher 重新搜索怎么办？
条件路由示例
def route_after_analyst(state: State) -> str:
    # 如果分析结果太短，质量不够
    if len(state["analysis"]) < 200:
        return "research"   # ← 回到 Researcher 重搜
    return "summariser"      # ← 正常继续

# 替换原来的 add_edge:
graph.add_conditional_edges(
    "analyst",
    route_after_analyst,
    {"research": "research", "summariser": "summariser"}
)
这就实现了自动重试循环，是 Reflexion / Self-Correction Agent 的基础模式。
▶️ 三种调用方式对比
invoke / stream / astream 用法
app = build_graph()

# ① invoke — 同步等待全部完成，返回最终 State

📌 扩展 1：添加 Fact-Checker Agent（核实环节）
在 Analyst 和 Summariser 之间插入一个 Fact-Checker，让它对 Analyst 的结论进行可信度评分。
新增 fact_checker_node
# 1. 扩展 State 添加新字段
class State(TypedDict):
    topic:       str
    research:    str
    analysis:    str
    fact_check:  str   # ← 新增
    summary:     str

# 2. 新 Agent 函数
def fact_checker_node(state: State) -> dict:
    groq_client = Groq(api_key=GROQ_API_KEY)
    response = groq_client.chat.completions.create(
        model=MODEL,
        messages=[{
            "role": "system",
            "content": "You are a fact checker. Review the analysis and rate each claim (HIGH/MEDIUM/LOW confidence). Flag unsupported claims."
        },{
            "role": "user",
            "content": f"Analysis:\n{state['analysis']}\n\nOriginal Research:\n{state['research']}"
        }]
    )
    return {"fact_check": response.choices[0].message.content}

# 3. 更新图（插入新节点）
graph.add_node("fact_checker", fact_checker_node)
graph.add_edge("analyst",      "fact_checker")   # ← 原来: analyst → summariser
graph.add_edge("fact_checker",  "summariser")
✅ 插入新节点只需：①加 State 字段 ②写函数 ③改两行 add_edge。这就是图编排的灵活性——流程修改不影响已有 Agent 的代码。
⚡ 扩展 2：流式输出（Streaming）
让用户看到每个 Agent 完成时的实时进度，而不是等到全部结束。
stream() 用法
def research_streaming(topic: str):
    app = build_graph()

    # stream() 在每个节点完成时 yield 该节点的输出
    for event in app.stream({"topic": topic}):
        # event 格式: {"node_name": {部分 State 更新}}
        for node_name, node_output in event.items():
            if node_name == "research":
                print("📚 搜索完成，找到文章...")
            elif node_name == "analyst":
                print("📊 分析完成，提炼见解...")
            elif node_name == "summariser":
                print("✅ 总结完成！\n")
                print(node_output["summary"])

# 如果要把 LLM token 级别的流式也透出来，
# 在 Groq 调用里加 stream=True：
response = groq_client.chat.completions.create(
    model=MODEL, messages=[...],
    stream=True            # ← 加这一行
)
for chunk in response:
    print(chunk.choices[0].delta.content or "", end="", flush=True)
💡 两种流式粒度：LangGraph 的 stream() 是节点级（每个 Agent 完成就推一次）；Groq 的 stream=True 是token 级（逐字推出）。根据 UX 需求选择。
🧠 扩展 3：记忆持久化（Checkpointer）
让研究助手"记住"之前的研究，支持断点续跑和多轮对话。
SQLite Checkpointer 示例
from langgraph.checkpoint.sqlite import SqliteSaver

# 创建持久化存储（SQLite 文件）
checkpointer = SqliteSaver.from_conn_string("research_memory.db")

# 编译时传入 checkpointer
app = graph.compile(checkpointer=checkpointer)

# 运行时传入 thread_id（对话 ID）
config = {"configurable": {"thread_id": "user_001_session_1"}}

result = app.invoke({"topic": "量子计算"}, config=config)

# 下次同一 thread_id 调用，LangGraph 会自动加载历史状态
# 可以从中断点继续，也可以查看历史
history = app.get_state_history(config)
for state in history:
    print(state.created_at, state.values.keys())
🔬 thread_id 就是"会话 ID"，同一用户的多次对话用同一 thread_id，就能实现连续记忆。这是 MemGPT 思想的 LangGraph 实现。
🧑‍💻 扩展 4：人工介入点（Human-in-the-Loop）
在 Analyst 完成后暂停，让人类确认分析质量，再决定是否继续生成报告。
interrupt_before 示例
# 编译时声明"在 summariser 节点前暂停"
app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["summariser"]          # ← 关键
)

config = {"configurable": {"thread_id": "review_001"}}

# 第一次调用：执行到 Analyst 后暂停
result = app.invoke({"topic": "AI安全"}, config=config)
# result 此时包含 research 和 analysis，但没有 summary

# 人类查看分析结果
current = app.get_state(config)
print("Analyst 结果：\n", current.values["analysis"])
approved = input("是否继续生成报告？(y/n): ")

if approved == "y":
    # 传 None 继续执行（从暂停点恢复）
    final = app.invoke(None, config=config)
    print(final["summary"])
⚡ 这就是生产 Agent 的"关键节点人工介入"模式。对高风险操作（发邮件、写数据库）前加 interrupt_before，让人类确认，大幅降低误操作风险。
⚡ 扩展 5：并行搜索（Map-Reduce 模式）
同时从多个数据源搜索，然后汇总结果，速度提升 N 倍。
并行节点示例
from typing import Annotated
import operator

class ParallelState(TypedDict):
    topic:      str
    # Annotated + operator.add 让多个并行节点的输出自动合并
    searches:   Annotated[list[str], operator.add]
    analysis:   str
    summary:    str

def search_web(state): ...      # 搜 Tavily
def search_arxiv(state): ...    # 搜 arXiv 论文
def search_wiki(state): ...     # 搜 Wikipedia

graph = StateGraph(ParallelState)
graph.add_node("search_web",   search_web)
graph.add_node("search_arxiv", search_arxiv)
graph.add_node("search_wiki",  search_wiki)
graph.add_node("analyst",      analyst_node)

# START → 三路并行
graph.add_edge(START,          "search_web")
graph.add_edge(START,          "search_arxiv")
graph.add_edge(START,          "search_wiki")

# 三路汇聚 → analyst（LangGraph 自动等待三路都完成）
graph.add_edge("search_web",   "analyst")
graph.add_edge("search_arxiv", "analyst")
graph.add_edge("search_wiki",  "analyst")
🔀 LangGraph 发现多条边同时指向 START 时，自动并行执行；发现多条边同时指向 analyst 时，自动等待所有上游完成（Join）。这是 Map-Reduce 模式，无需手写 asyncio.gather。
